**Verdict:** план пока не готов к реализации без правок. Он покрывает все 10 исходных замечаний по списку, но несколько решений закрывают проблему только декларативно или создают новые регрессии.

**Основные проблемы плана**

1. **P1-2 / P3-8: `tee` нельзя оставлять с замаскированным exit code как "некритично".**
   В плане предлагается `claude ... 2>&1 | tee -a "$session_log"` и прямо говорится, что код возврата `claude` маскируется. Это не просто комментарий: быстрый failure будет виден в логе, но launcher/tmux получит успех от `tee`, что ухудшает диагностику завершения. В launcher надо явно сохранять статус левой части pipeline: `set -o pipefail` в bash или `exit "${PIPESTATUS[0]}"`.

2. **P3-8 переоценивает "argv-style" совместимость с tmux.**
   `tmux new-session` все равно принимает shell-command, который запускается через shell. Формулировка "tmux зовет его как `claude-control-run "$path" ...`" не устраняет shell layer полностью. Launcher полезен, но план должен честно сказать: shell-строка остается, просто становится короче и квотит меньше аргументов. И обязательно проверить путь к launcher, `path` с пробелами, quotes и newline/control chars, иначе исходный P3-8 закрыт частично.

3. **P2-4: logrotate описан слишком размыто и не покрывает все реальные логи.**
   План перечисляет `control.log`, `control.err`, `watchdog.log`, `sessions/*.log`, но в установке есть еще `watchdog.out` и `watchdog.err` в launchd/systemd templates. Если критерий "все логи" остается, их надо включить или явно исключить.
   Также "copytruncate-семантика" не решает race, который был в аудите: строки между `tail` и truncate/write всё еще могут потеряться/перемешаться. Это допустимо как tradeoff, но тогда критерий готовности "ни один лог не растет безлимитно" должен быть дополнен тестом и явным отказом от гарантий сохранности строк при live write.

4. **P2-4 и `--no-watchdog`: порядок установки недоописан.**
   Сейчас `--no-watchdog` физически удаляет watchdog unit/timer. План говорит, что logrotate должен быть независим от health-watchdog, но не фиксирует поведение при `--no-watchdog`: новый logrotate timer должен ставиться всё равно, иначе исходная проблема "при `--no-watchdog` логи безлимитны" останется.

5. **P2-5: строгий heartbeat рискованнее, чем указано.**
   Переход от "имя session видно" к "есть healthy-marker из `claude-rc status`" может резко увеличить false negative. Маркеры `capacity:`, `reconnected after`, `show qr code` могут не присутствовать в последних 30 строках `control.log`, быть transient, попасть под ANSI/redraw или отличаться для control-сессии. Тогда watchdog будет делать лишний `kick` каждые 5 минут. Плану нужен более безопасный дизайн: strip ANSI, большее окно, несколько последовательных misses, last-relevant-status-event как в P1-1, и fallback/градация вместо одного жесткого grep.

6. **P1-1: "последняя строка лога" для dead session сформулирована опасно.**
   Сказано применить логику к "последней строке лога", но правильный вариант: последняя релевантная статусная строка/событие. Последняя физическая строка может быть redraw, prompt, пустая строка или мусор после настоящей ошибки.

7. **P3-9: `CLAUDE_BIN` требует правки install/prereq, не только двух скриптов.**
   Код сейчас проверяет `command -v claude` в `install.sh`, а `claude-control-session` исполняет literal `claude`. Если реально поддерживать `CLAUDE_BIN`, install должен проверять `${CLAUDE_BIN:-claude}` или хотя бы не падать при отсутствии `claude`, когда задан `CLAUDE_BIN`. На Linux env file есть, на launchd env file нет, значит macOS-путь тоже надо явно продумать.

8. **P1-3 обновляет example, но не существующий установленный `~/.claude-control/CLAUDE.md`.**
   `install.sh` сохраняет существующий файл. Поэтому усиление `examples/control-CLAUDE.md.example` не дойдет до текущих пользователей. План должен добавить migration note/вывод после install или отдельный doc-шаг.

**Что поправить в плане до старта**

- Для launcher: указать `bash`, `set -o pipefail`, возврат статуса `claude`, установку нового `bin/claude-control-run` через `install.sh`, и честно описать оставшийся tmux shell-command layer.
- Для logrotate: перечислить все файлы (`watchdog.out/err` тоже), определить поведение при `--no-watchdog`, запретить rename для live-written логов, зафиксировать race как accepted limitation и добавить тесты таймеров/юнитов.
- Для heartbeat: не делать сразу жесткий AND по имени и healthy-marker в последних 30 строках. Нужен более мягкий алгоритм с ANSI-strip, большим окном и несколькими неудачными тиками до kick.
- Для `CLAUDE_BIN`: включить `install.sh`, launchd/systemd env story и prereq checks.
- Для P1-1: заменить "последняя строка лога" на "последнее релевантное статусное событие".

После этих правок план будет хорошей основой для реализации. Сейчас он полезный, но часть P1/P2 рисков в нем недооценена.
