## Часть 1. Второе мнение Codex (дословно)

Я бы шел не "принципиально иначе", а **сильно урезанным вариантом этого плана**, с учетом 8 замечаний ревью. Архитектура для личного инструмента нормальная: `control + supervisor + tmux + watchdog` достаточно простая и понятная. Но полный план уже начинает превращать маленькую Unix-надстройку в мини-сервис с отдельными таймерами, миграциями и полупротоколом поверх TUI. Это не лучший первый шаг.

**MVP, который я бы сделал первым:**

1. Исправить `claude-rc status`: классификация по **последнему релевантному статусному событию**, а не grep по всему хвосту; одинаково для live pane и dead log. Сразу добавить ANSI-strip, quoted/untrusted evidence и проверку проекта через `projects.yaml`.

2. Убрать race логирования проектных сессий: маленький launcher, лог с первого байта, `bash`, `set -o pipefail`, возврат `${PIPESTATUS[0]}`. Туда же реально протянуть `CLAUDE_BIN`, но не забыть `install.sh`/prereq/env story.

3. Закрыть простые sharp edges: валидация `SESSION_NAME`, честный комментарий про traversal-check, migration note для уже установленного `~/.claude-control/CLAUDE.md`.

4. Логи: я бы **не начинал с отдельного launchd/systemd logrotate timer**. Для личного инструмента это перебор на первом проходе. Лучше сделать один `claude-control-logrotate`/`trim_logs` и вызывать его оппортунистически из watchdog и `claude-rc` start/status. Если потом реально увидишь рост `control.log` при `--no-watchdog`, тогда добавить отдельный timer.

Что бы сознательно отложил: строгий новый heartbeat, отдельный logrotate timer, глубокую поддержку экзотических путей с newline/control chars, "полную" atomic-ротацию live-written логов. Особенно heartbeat: замена "вижу имя control" на жесткий healthy-marker может ухудшить систему лишними рестартами. Если трогать, то мягко: больше окно, strip ANSI, несколько misses подряд, не один grep.

По уровню ниже TUI: я проверил локальный Claude Code `2.1.156`. Публичная справка показывает `--remote-control`, но не показывает `remote-control status`/`--json`. `claude agents --json` есть, но это про background agents, не про remote-control. В bundled binary виден скрытый слой `claude daemon remote-control list --json`, но в этой среде команда отвечает: `'daemon remote-control' is not available in this environment.` То есть **перспективный машинный источник состояния есть в недрах CLI, но сегодня на него нельзя надежно опереться как на публичный контракт**.

Итоговая рекомендация: идти этим курсом, но не реализовывать весь план как один "аудит-комплаенс" пакет. Сначала сделать маленький надежный слой: правильный `status`, раннее логирование, `CLAUDE_BIN`, валидации, untrusted evidence. После этого отдельно сделать spike: можно ли в твоей реальной logged-in среде использовать `claude daemon remote-control list --json`. Если да, переводить health/status на него. Если нет, оставить TUI-парсинг как явно эвристический и не усложнять его сверх MVP.

## Часть 2. Проверка машинного источника состояния (локально, среда пользователя)

Версия: `claude` 2.1.156 (Claude Code). Все команды read-only.

Итог: **публичного машиночитаемого контракта состояния для `remote-control` сегодня нет** - подтверждено в реальной среде, не только в песочнице Codex.

Что проверено:

- `claude daemon remote-control list --json` -> `'daemon remote-control' is not available in this environment.` (та же недоступность, что у Codex).
- `claude remote-control --help` - никаких `--json` или status-опций. Интерфейс сессии - только TUI ("Remote Control runs as a persistent server").
- `claude daemon status` - существует, но это про *daemon*-слой:
  - есть `control.sock` (`/tmp/cc-daemon-501/<id>/control.sock`) и `roster.json` со списком bg-workers;
  - сейчас `not running`, `control.sock: unreachable`, `roster.json: absent`;
  - `--json` для `daemon status` игнорируется (`warning: extra arguments ignored: --json`).
- `claude daemon --help` - сам daemon "runs on demand and exits when the last client disconnects", service install отключен в этой версии.

Вывод для проекта: сессии, которые `claude-rc` поднимает через `claude remote-control --name ...` в tmux, через `claude daemon` НЕ идут (общего `roster.json` нет). Опереться на машинный сигнал нельзя - парсинг TUI остается единственным источником, и его надо подавать честно как эвристику.

Будущий spike (когда/если станет публичным): если в реальной залогиненной среде заработает `claude daemon remote-control list --json` (или появится `roster.json`/`control.sock` для remote-control сессий) - перевести health/status на него и снять целый класс TUI-замечаний. Пока - не закладываться.
